# 03 - 整体架构

本文档从宏观视角解释 JHarness 的分层设计、启动装配与核心执行流。

## 1. 分层架构

JHarness 采用清晰的 **五层架构**，每层职责单一、向下依赖：

```
┌────────────────────────────────────────────────────────┐
│                     Presentation                       │  ui/, command/
│  ConsoleInteractiveSession  │  TerminalUI(Lanterna)    │
│  CommandRegistry (50+ slash commands) │  Keybindings   │
├────────────────────────────────────────────────────────┤
│                     Application                        │  JHarnessApplication
│  Picocli CLI  │  装配 (buildQueryEngine)  │ 生命周期   │
├────────────────────────────────────────────────────────┤
│                     Orchestration                      │  core/engine/, agent/
│  QueryEngine (ReAct)  │  AgentOrchestrator  │  Hooks   │
│  MessageCompaction  │  BackgroundTaskManager  │ Cron   │
├────────────────────────────────────────────────────────┤
│                       Domain                           │  tools/, session/, extension/
│  Tools(40+)  │  PermissionChecker  │  Plugins/Skills   │
│  SessionStorage  │  MemoryManager                      │
├────────────────────────────────────────────────────────┤
│                     Integration                        │  integration/
│  OpenAiApiClient(SSE)  │  McpClientManager  │  Cron    │
│  RetryPolicy  │  Http/Stdio Transport                  │
└────────────────────────────────────────────────────────┘
        │                    │                   │
        ▼                    ▼                   ▼
    LLM Provider       MCP Servers          External Systems
```

### 分层职责

| 层 | 包 | 职责 |
|----|-----|-----|
| **Presentation** | `ui/`、`command/` | 用户交互界面、斜杠命令分发、快捷键解析 |
| **Application** | `JHarnessApplication` | CLI 参数解析、组件装配、进程生命周期 |
| **Orchestration** | `core/engine/`、`agent/` | ReAct 循环、多 Agent 编排、生命周期 Hook |
| **Domain** | `tools/`、`session/`、`extension/`、`core/` | 业务能力：工具、权限、会话、扩展、状态 |
| **Integration** | `integration/` | 与 LLM / MCP 服务器 / Cron 等外部系统的协议实现 |

## 2. 启动装配流程

应用入口 `JHarnessApplication#call()` 是整个依赖图的装配中心。其 `buildQueryEngine` 方法清晰展示了各组件的组合顺序：

```
Settings.load()                          // 1. 环境变量 + ~/.jharness/settings.json
    │
    ▼
OpenAiApiClient(baseUrl, apiKey, model, maxTokens)   // 2. LLM 传输层
    │
    ├─ BackgroundTaskManager(dataDir/tasks)          // 3a. 后台任务
    ├─ TeamRegistry()                                // 3b. 多智能体团队
    ├─ SkillRegistry.withProject(cwd)                // 3c. 技能注册
    ├─ McpClientManager()                            // 3d. MCP 客户端
    └─ CronRegistry(dataDir)                         // 3e. Cron 注册
    │
    ▼
ToolRegistry.withDefaults(...)                       // 4. 默认 40+ 工具 + MCP 动态工具
    │
    ▼
PermissionChecker(mode)                              // 5. 权限栅栏
    ├─ allowedTools / deniedTools
    ├─ pathRules (realPath + 符号链接解析)
    └─ deniedCommandPatterns (token 归一化 + glob)
    │
    ├─ 注入回 BackgroundTaskManager                  // FP-3 共享安全栅栏
    └─ 注入回 McpClientManager
    │
    ▼
SystemPromptBuilder.build()                          // 6. 系统提示词（可注入 CLAUDE.md）
    │
    ▼
QueryEngine(apiClient, toolRegistry, checker, cwd, systemPrompt, maxTurns)
    │
    ├─ costTracker.setModelName(model)               // F-P0-5 成本估算
    └─ AgentTool.configureInProcess(...)             // F-P1-1 子 Agent 使用同一进程
```

关键安全设计：**PermissionChecker 是共享单例**。前台 `BashTool`、后台 `BackgroundTaskManager` 与 `McpClientManager` fork 子进程前全部复用同一个 checker，避免黑名单 / 路径规则通过后台通道被绕过（代码注释中的 **FP-3**）。

## 3. 核心执行流（ReAct 循环）

单次用户查询在 `QueryEngine#submitMessage → runQuery` 中的执行过程：

```
 ┌─────────────────────────────────────────────────────────┐
 │ submitMessage(prompt, eventConsumer)                    │
 │  ├─ cancelled.set(false)                                │
 │  └─ messages.add(userText)                              │
 └─────────────────────────────────────────────────────────┘
                        │
                        ▼
        ┌──────────────────────────────┐
        │   runQuery (loop maxTurns)    │  ◄──── 最大 maxTurns 轮
        └──────────────────────────────┘
          │
          │ 1. 每轮开始：检查 cancelled、压缩消息
          ▼
   ┌──────────────────────────────────────┐
   │ apiClient.streamMessage(messages,    │
   │   systemPrompt, toolsSchema,         │
   │   eventConsumer)                     │
   │   ← SSE 流式解析                     │
   └──────────────────────────────────────┘
          │
          │ 发射: AssistantTextDelta / ToolExecutionStarted ...
          │ 最终返回: ApiMessageCompleteEvent { blocks, usage }
          ▼
   ┌──────────────────────────────────────┐
   │ costTracker.addUsage(usage)          │  ← 超预算抛 BudgetExceededException
   └──────────────────────────────────────┘
          │
          ▼
   ┌──────────────────────────────────────┐
   │ 解析响应 blocks                      │
   │  ├─ TextBlock  → 追加到对话历史      │
   │  └─ ToolUseBlock → 收集进 toolCalls  │
   └──────────────────────────────────────┘
          │
          │ 若 toolCalls 为空 → 退出循环（turn complete）
          ▼
   ┌──────────────────────────────────────┐
   │ executeTools(toolCalls, consumer)    │
   │                                      │
   │ 对每个 ToolUseBlock 并行执行：        │
   │  1. PermissionChecker.evaluate()     │  ← DENY → 立即失败
   │  2. BaseTool.execute(input, ctx)     │  ← CompletableFuture
   │  3. 封装 ToolResultBlock             │
   │                                      │
   │ PLAN 模式 → interceptToolCallsForPlan │  ← 改为生成 ExecutionPlan
   └──────────────────────────────────────┘
          │
          ▼
   ┌──────────────────────────────────────┐
   │ messages.add(assistantMsg)           │
   │ messages.add(toolResultsMsg)         │
   └──────────────────────────────────────┘
                        │
                        └─────── 回到第 1 步，进入下一轮
```

### 流式事件体系

`core/engine/stream/` 下定义了 5 种流式事件（均实现 `StreamEvent` 标记接口）：

| 事件 | 触发时机 | 用途 |
|-----|---------|-----|
| `AssistantTextDelta` | LLM 输出文本增量 | 实时打字机渲染 |
| `ToolExecutionStarted` | 工具开始执行 | UI 显示「🔧 执行工具: xxx」 |
| `ToolExecutionCompleted` | 工具完成（含成功/失败） | UI 显示 ✅ / ❌ |
| `AssistantTurnComplete` | 一轮 LLM 调用结束 | UI 换行、打印提示语 |
| `UsageReport` | Token 用量报告 | 成本面板刷新 |

上层（CLI / TUI）通过 `Consumer<StreamEvent>` 订阅事件并作出渲染。

## 4. 消息模型

`core/engine/model/` 定义了与 LLM 协议对齐的内容块模型：

```
ConversationMessage
  ├─ role: MessageRole { USER, ASSISTANT, SYSTEM, TOOL }
  └─ content: List<ContentBlock>
                │
                ├─ TextBlock          ── 纯文本
                ├─ ImageBlock         ── 图像（base64 / url）
                ├─ ToolUseBlock       ── 模型请求调用工具
                └─ ToolResultBlock    ── 工具执行结果回写
```

这种「分块内容」设计使得同一条消息中可以混排文字、图片与工具调用，贴合 Anthropic Messages API / OpenAI Tool Calls 的最新形态。

## 5. 安全栅栏与审计

```
         Tool Call
             │
             ▼
   ┌──────────────────────┐
   │ PermissionChecker    │
   │  evaluate()           │
   └──────────────────────┘
     1. deniedTools          ── 全模式强制
     2. deniedCommands       ── 全模式，token 归一化
     3. pathRules            ── 全模式，realPath 解析
     4. allowedTools         ── 仅跳过确认
     5. mode 决策：
          DEFAULT    → 只读放行 / 其他需要确认
          FULL_AUTO  → 全部放行（但前 1-3 步仍执行）
          PLAN       → 非只读一律拒绝
             │
             ▼
   ┌──────────────────────┐
   │ auditLogger.info(...) │  ── 独立 Logger
   └──────────────────────┘
             │
             ▼
   PermissionDecision { allow / deny / requiresConfirmation }
```

详见 [08-权限系统](08-权限系统.md)。

## 6. 资源生命周期

JHarness 严格使用 `AutoCloseable`：

| 组件 | 资源 | 关闭时机 |
|------|-----|---------|
| `QueryEngine` | API 客户端、HTTP 池、线程池 | `try-with-resources` 于 `runPrintMode` / `runInteractiveMode` |
| `OpenAiApiClient` | OkHttp Dispatcher、`activeEventSources`、`activeFutures` | `QueryEngine#close` 级联 |
| `BackgroundTaskManager` | `jharness-bg-task-N` 固定线程池、子进程 | CLI 退出或显式关闭 |
| `McpClientManager` | stdio 子进程、HTTP 连接、线程池 | `close()` 幂等 |
| `AgentOrchestrator` | `jharness-agent-N` 有界线程池 | `shutdown()` |
| `CronRegistry` | `jharness-cron-N` daemon 调度池 | JVM 退出自动清理（daemon） |

**异常退出保障**：`reportFatal` 会把根因打印到 stderr 并写 `logger.error`，便于脚本解析；`try-with-resources` 保证即使抛出 `BudgetExceededException` 等也不会泄漏 HTTP 连接。

## 7. 并发模型

| 并发原语 | 使用位置 | 目的 |
|---------|---------|-----|
| `ConcurrentHashMap` | `ToolRegistry.tools`、`BackgroundTaskManager.tasks`、`McpClientManager.sessions` | 注册表读写 |
| `CopyOnWriteArrayList/Set` | `PermissionChecker.pathRules/deniedCommands/allowedTools` | 读多写少、迭代期间允许修改 |
| `AtomicBoolean` | `QueryEngine.cancelled` | 取消标志 |
| `AtomicLong/Integer/Reference` | `CostTracker` 所有计数与金额 | 无锁累计 |
| `synchronized(messageLock)` | `QueryEngine.messages` | 读写同步，避免 `synchronizedList` + 独立锁双重同步死锁 |
| `ThreadPoolExecutor` + `CallerRunsPolicy` | `AgentOrchestrator`、`McpClientManager` | 有界回压，避免线程爆炸 |
| `ScheduledExecutorService` (daemon) | `CronRegistry` | 定时调度 |
| `CompletableFuture.allOf` | 多 Agent 并行 | 聚合结果 |
