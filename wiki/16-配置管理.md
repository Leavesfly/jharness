# 16 · 配置管理

> 包路径：`io.leavesfly.jharness.config`
>
> 核心类：
> - `Settings` — 配置 POJO + 多层加载 / 持久化
> - `DefaultSettingsLoader` — classpath `defaults/settings.json` 加载器（带缓存）
> - `SettingsBootstrap` — 首启 seed 默认资源到 `~/.jharness/`
> - `OnboardingMarker` — `~/.jharness/.onboarded` 标记文件管理

---

## 1. 配置加载优先级（高优先级覆盖低优先级）

```
1. CLI 选项 (-m / --max-turns / --permission-mode ...，最高优先级)
2. 环境变量 (OPENAI_API_KEY / JHARNESS_MODEL 等)
3. 用户配置 (~/.jharness/settings.json)
4. classpath 默认 (defaults/settings.json，由 DefaultSettingsLoader 加载)
5. 字段硬编码兜底 (Settings 构造器初始值，最低优先级)
```

`Settings.load()` 内部按从低到高顺序逐层 `mergeFromJson(JsonNode)` 增量覆盖，规则：

- 引用类型字段：仅当 JSON 节点 `hasNonNull` 时才覆盖；
- primitive 字段：仅当节点 `has` 且类型匹配时才覆盖（避免 Jackson 默认 `0/false` 误覆盖）；
- 集合 / Map 字段：整体替换（不做深度合并）；
- 环境变量在文件加载后再叠加一次（`loadFromEnvironment()`），保证 `export OPENAI_*` 永远生效。

> **关于项目级配置**：当前实现中 `Settings.load()` 仅读取 `~/.jharness/settings.json`，**未自动合并 `<cwd>/.jharness/settings.json`**。如需项目级覆盖，请通过 `CLAUDE.md`（项目上下文）或 CLI 选项实现，或在调用方手动构造项目级合并逻辑。

## 3. 配置文件示例

`~/.jharness/settings.json`（完整示例）：

```json
{
  "model": "deepseek-chat",
  "baseUrl": "https://api.deepseek.com/v1",
  "maxTokens": 8192,
  "maxTurns": 12,
  "permissionMode": "DEFAULT",

  "effort": "medium",
  "passes": 1,
  "theme": "dark",
  "vimEnabled": false,
  "fastMode": false,
  "outputStyle": "default",

  "dailyBudgetUsd": 5.0,
  "connectTimeoutSeconds": 30,
  "readTimeoutSeconds": 300,
  "writeTimeoutSeconds": 30,

  "messageCompactionTokenBudget": 32000,
  "messageCompactionMaxMessages": 20,
  "autoSaveSessions": true,

  "allowedTools": ["read_file", "glob", "grep", "lsp"],
  "deniedTools": [],
  "pathRules": [
    { "pattern": "**/.git/**", "allow": false },
    { "pattern": "**/.env*",   "allow": false },
    { "pattern": "src/**",      "allow": true }
  ],
  "deniedCommandPatterns": [
    "rm -rf /*",
    "sudo *",
    "curl * | sh"
  ],

  "enabledPlugins": {
    "my-plugin": true,
    "another-plugin": false
  },

  "mcpServers": {
    "filesystem": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"]
    }
  }
}
```

**API Key 不写入文件**：`Settings.apiKey` 字段用 `@JsonIgnore` 标记，避免 `toJson` / `save` 时泄漏到磁盘。Key 只能通过环境变量或 `/login` 命令（写入独立的凭据存储）提供。

## 5. 默认资源 seed（`SettingsBootstrap`）

首次启动且 `~/.jharness/settings.json` 不存在时，从 classpath 释放默认资源到用户主目录：

| classpath 资源 | 目标路径 |
|---------------|---------|
| `defaults/settings.json` | `~/.jharness/settings.json` |
| `defaults/plugins/builtin-defaults/` | `~/.jharness/plugins/builtin-defaults/` |

### 5.1 设计要点

1. **幂等**：仅在目标不存在时拷贝，绝不覆盖用户修改
2. **失败不阻断**：任何 IO 异常降级为 warn 日志，主流程继续
3. **兼容两种部署**：同时支持 classpath 为 `file:` 目录（IDE 调试）和 `jar:` 归档（生产 JAR）

### 5.2 入口

```java
SettingsBootstrap.seedIfAbsent();              // 用默认 ~/.jharness
SettingsBootstrap.seedIfAbsent(customDir);     // 测试友好
```

`JHarnessApplication.call()` 在 `Settings.load()` 之前调用一次。

### 5.3 `DefaultSettingsLoader`（运行时加载，不写盘）

与 `SettingsBootstrap`（一次性 seed 到用户目录）不同，`DefaultSettingsLoader` 每次 `Settings.load()` 调用时都会读取 classpath `defaults/settings.json` 作为「应用级默认值层」打底：

```java
// 由 Settings.load() 内部调用
DefaultSettingsLoader.loadClasspathDefaults()
        .ifPresent(settings::mergeFromJson);

// 用户配置文件
DefaultSettingsLoader.loadFromPath(Settings.getDefaultConfigDir().resolve("settings.json"))
        .ifPresent(settings::mergeFromJson);
```

特性：

- **进程内缓存**：classpath 资源在进程生命周期内不变化，首次解析后缓存 `JsonNode`，避免反复 IO。
- **缺失软降级**：classpath 资源不存在 / JSON 根节点非对象 / 解析失败时，返回 `Optional.empty()` 并 warn，调用方继续走下层默认值。
- **测试友好**：暴露 `loadFromPath(Path)` 与包内 `resetCacheForTesting()`，便于在单测中注入临时文件。

## 5.4 Onboarding 与 `.onboarded` 标记

首次启动除了 seed 资源，还可能触发 **Agent Onboarding 向导**（详见 [`02-快速开始 §3`](02-快速开始.md#3-首次启动)）：

| 角色 | 包路径 | 职责 |
|------|--------|------|
| `app.onboarding.OnboardingService` | `app/onboarding/` | 交互式向导（Provider 选择 / Base URL / Model / API Key 脱敏输入 / 权限模式），结果就地修改 `Settings` 并合并写入 `~/.jharness/settings.json` |
| `config.OnboardingMarker` | `config/` | 管理 `~/.jharness/.onboarded` 标记文件的存在性检查、写入、删除（下沉到 `config` 包以满足 ArchUnit 守卫：`command` 不允许依赖 `app`） |

### 触发条件（`OnboardingService#shouldRun`）

必须同时满足：

1. `~/.jharness/.onboarded` 标记文件不存在；
2. 当前 `Settings` 处于"无法直接可用"的状态——`apiKey` 为空 **且** `baseUrl` 非本地端点。

若 `apiKey` 已配置或 `baseUrl` 指向本地 Ollama，向导**静默跳过**并写入 marker 避免后续打扰。

### CLI 触发判定（`JHarnessApplication#maybeRunOnboarding`）

在 `Settings.load()` 之后、API Key 校验之前调用，需全部满足：

- 未传 `--skip-onboarding`；
- 未传 `-p/--print`（非单次查询）；
- `--output-format` 为 `text`（避免污染 json / stream-json）；
- `System.console() != null`（真实终端，非管道 / 重定向 / CI）；
- `OnboardingService#shouldRun` 返回 `true`。

### `apiKey` 持久化注意事项

`Settings#getApiKey` 标记了 `@JsonIgnore`，直接 `toJson` / `save` 不会写入 apiKey。Onboarding 通过手动合并策略写入：

```java
ObjectNode node = PRETTY_MAPPER.valueToTree(settings);
if (apiKey != null && !apiKey.isBlank()) {
    node.put("apiKey", apiKey);
}
Files.writeString(settingsJson, PRETTY_MAPPER.writeValueAsString(node), CREATE, TRUNCATE_EXISTING);
```

这是 onboarding 场景唯一一处会把 apiKey 落到 `settings.json` 的地方，其他场景（`/config set` / `Settings.save()`）仍然遵守 `@JsonIgnore` 约定，避免普通保存路径意外覆盖把 apiKey 写出去。

### Marker 管理 API

```java
OnboardingMarker.isOnboarded();                  // 默认目录是否已完成
OnboardingMarker.markOnboarded();                // 写 marker
OnboardingMarker.clearMarker();                  // 删 marker（/onboarding 命令用此）
OnboardingMarker.isOnboarded(customConfigDir);   // 指定目录（测试用）
```

## 6. 关键 API

### 6.1 读 / 写

```java
Settings s = Settings.load();          // 多层加载
s.setModel("gpt-4o");
s.setDailyBudgetUsd(new BigDecimal("10.0"));
s.save();                              // 写回 ~/.jharness/settings.json（不含 apiKey）

String json = s.toJson();              // 序列化为 JSON 字符串
```

### 6.2 动态 get / set

```java
String value = s.get("model");                          // → "qwen3.5:4b"
boolean ok = s.set("permissionMode", "FULL_AUTO");      // → true
boolean ok2 = s.setPermissionMode("plan");              // 大小写不敏感
```

支持的 key：`model` / `apiKey` / `baseUrl` / `maxTokens` / `permissionMode` / `maxTurns` / `theme` / `effort` / `passes` / `fastMode` / `vimEnabled` / `voiceEnabled` / `outputStyle` 等。

### 6.3 路径常量

```java
Settings.getDefaultConfigDir()    // → ~/.jharness
Settings.getDefaultDataDir()      // → ~/.jharness/data
```

## 7. 子配置文件

JHarness 把不同领域的配置拆到独立文件，便于版本管理：

| 文件 | 内容 | 加载方 |
|------|------|--------|
| `~/.jharness/settings.json` | 主配置 | `Settings.load()` |
| `~/.jharness/hooks.json` | Hook 注册 | `HookRegistry` |
| `~/.jharness/mcp-servers.json` | MCP 服务器 | `McpClientManager.addServer` |
| `~/.jharness/keybindings.json` | 按键绑定 | `KeybindingLoader` |
| `~/.jharness/skills/*.md` | 用户技能 | `SkillLoader` |
| `~/.jharness/plugins/*/plugin.json` | 用户插件 | `PluginLoader` |
| `~/.jharness/data/sessions/*.json` | 会话快照 | `SessionStorage` |
| `~/.jharness/data/memories/<project>/*.json` | 跨会话记忆 | `MemoryManager` |
| `~/.jharness/data/cron_jobs.json` | 定时任务 | `CronRegistry` |
| `~/.jharness/data/tasks/<id>.log` | 后台任务输出 | `BackgroundTaskManager` |
| `~/.jharness/plugin-trust.json` | 插件信任 | `extension/plugins/trust/` |

项目级同名文件位于 `<cwd>/.jharness/` 下，优先级高于用户级。

## 9. 项目级配置

`<cwd>/.jharness/settings.json` 用于项目级覆盖。常见用法：

```json
{
  "model": "deepseek-chat",
  "permissionMode": "PLAN",
  "allowedTools": ["read_file", "glob", "grep"],
  "systemPrompt": "你是一个 Java 微服务架构专家。"
}
```

加载顺序：用户级 → 项目级（项目级 merge 覆盖用户级）。

### 9.1 系统提示词文件

除 `settings.systemPrompt` 外，还可以放独立的 `CLAUDE.md`（兼容 Claude Code 生态）或 `JHARNESS.md` 作为系统提示词文件：

```
<cwd>/CLAUDE.md
<cwd>/JHARNESS.md
~/.jharness/CLAUDE.md
```

加载顺序：`Settings.systemPrompt` > 项目 `CLAUDE.md` > 用户 `CLAUDE.md` > 内置默认。

`MessageCompactionService.withSystemPromptTokens(...)` 会自动扣除系统提示词的 token 预算。

## 10. 安全注意事项

1. **API Key 不写文件**：通过 `@JsonIgnore` 隔离
2. **路径规则 / 命令黑名单** 是项目级共享的安全栅栏，不要在公开仓库提交项目级 `settings.json` 时泄漏内部规则
3. **MCP 服务器配置** 中的 `headers.Authorization` 等敏感字段同样应通过环境变量注入
4. **`/login` 写的凭据** 默认放 `~/.jharness/credentials.json`，权限 0600
