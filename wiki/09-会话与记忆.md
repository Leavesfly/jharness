# 09 · 会话与记忆

> 包路径：
> - 会话：`io.leavesfly.jharness.capability.session`
> - 记忆：`io.leavesfly.jharness.kernel.memory`

JHarness 把"短期上下文"和"长期知识"分成两套独立机制：

- **会话（Session）** — 一次连续的对话流，含完整消息历史 + 工具调用 + token 用量，按 ID 持久化
- **记忆（Memory）** — 跨会话的结构化知识条目，按项目 / 分类组织，支持搜索

---

## 1. 会话快照（SessionSnapshot）

### 1.1 数据模型

```java
public class SessionSnapshot {
    String sessionId;          // 8 位 UUID 前缀
    String cwd;                // 创建时的工作目录
    String model;              // 使用的模型名
    List<ConversationMessage> messages;
    UsageSnapshot usage;       // 累计 token 用量
    Instant createdAt;
    String summary;            // 可选摘要
    int messageCount;
}
```

`UsageSnapshot` 含 `inputTokens` / `outputTokens` / `cacheReadInputTokens` / `cacheCreationInputTokens`。

### 1.2 文件布局

```
~/.jharness/data/sessions/
├── abc12345.json       # 单个会话快照（JSON Pretty）
├── def67890.json
├── session-tmp-xxx.tmp # 原子写中途残留（启动时自动清理）
└── ...
```

## 2. `SessionStorage`

```java
public class SessionStorage {
    public SessionStorage(Path sessionsDir);            // 启动时自动清理 .tmp 残留
    public void saveSession(SessionSnapshot snap);      // 原子写：先写 .tmp 再 ATOMIC_MOVE
    public SessionSnapshot loadSession(String sessionId);
    public List<SessionSnapshot> listSessions();         // 按 createdAt 倒序
    public boolean deleteSession(String sessionId);
}
```

### 2.1 原子写与并发安全

每个 `sessionId` 共享一把 **静态 per-id 锁**：

```java
private static final Map<String, Object> SAVE_LOCKS = new ConcurrentHashMap<>();
```

- 锁 key = `"绝对路径#sessionId"`（跨目录同名 ID 也安全）
- **锁对象常驻不 remove**（如果 finally 中 remove，会让两个线程 computeIfAbsent 拿到不同实例失去互斥）
- 同步块内使用 `Files.move(tmp, target, StandardCopyOption.ATOMIC_MOVE)` 保证写入原子性

### 2.2 ObjectMapper 配置

```java
new ObjectMapper()
    .registerModule(new Jdk8Module())
    .registerModule(new JavaTimeModule())
    .disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS);
```

`Instant` 序列化为 ISO-8601 字符串，便于人读。

## 3. 自动保存

由 `QueryEngineBuilder.configureAutoSaveSession()` 配置（`autoSaveSessions=true` 默认开启）：

```java
engine.setSessionPersister(messages -> {
    SessionSnapshot snap = new SessionSnapshot(
        sessionId, cwd.toString(), model,
        messages, engine.getCostTracker().toUsageSnapshot(),
        createdAt, /* summary */ null, messages.size());
    storage.saveSession(snap);
});
```

`QueryEngine` 在每轮消息变更（压缩 / 工具结果 / 异常 / 取消）后调用一次 `persistIfNeeded`，异常被吞掉只记 warn，保证主循环不受影响。

## 4. 恢复会话

### 4.1 CLI 入口

```bash
# 继续上一次会话（按 createdAt 找最新）
java -jar jharness.jar -c

# 恢复指定会话
java -jar jharness.jar -r abc12345
```

由 `app.cli.SessionRestorer` 处理：

```java
SessionSnapshot snap = storage.loadSession(sessionId);
engine.loadMessages(snap.getMessages());
// usage 不复用（避免重复计费），仅作信息展示
```

### 4.2 交互式命令

| 命令 | 行为 |
|------|------|
| `/resume` | 列出最近会话，按编号恢复 |
| `/session list` | 列出全部 |
| `/session switch <id>` | 切换 |
| `/rewind <n>` | 回滚到第 n 条消息（用于"撤销最近几轮对话"） |
| `/compact` | 立即触发消息压缩 |
| `/export` | 导出当前会话为纯文本 |
| `/share` | 生成分享链接 / 标识 |
| `/tag` | 给会话打标签 |
| `/summary` | 让 LLM 生成当前会话摘要 |

## 5. 记忆管理（MemoryManager）

### 5.1 数据模型

```java
public class MemoryEntry {
    String title;
    String content;
    String category;           // 可选分类
    LocalDateTime createdAt;
    LocalDateTime updatedAt;
}
```

### 5.2 文件布局

```
~/.jharness/data/memories/
├── jharness/              # project 名（sanitize 后的目录名）
│   ├── 项目架构.json
│   ├── 编码规范.json
│   └── ...
├── another-project/
│   └── ...
└── global/                # 全局记忆
    └── ...
```

按 `project` 字段隔离，避免不同项目记忆混淆。`sanitize(project)` 防止路径穿越。

### 5.3 API

```java
public class MemoryManager {
    public MemoryManager(Path memoryDir);

    // CRUD
    public void addMemory(String project, String title, String content);
    public void addMemory(String project, String title, String content, String category);
    public boolean updateMemory(String project, String title, String content);
    public boolean removeMemory(String project, String title);
    public void clearProjectMemories(String project);

    // 查询
    public List<String> listMemories(String project);
    public MemoryEntry getMemoryDetails(String project, String title);
    public String readMemory(String project, String title);
    public List<String> searchByCategory(String project, String category);
    public List<String> searchMemories(String project, String keyword);   // 全文搜索

    // 统计 / 清理
    public MemoryStats getStats(String project);   // count + size + 分类计数
    public int cleanupOldMemories(String project, int daysOld);
}
```

### 5.4 `/memory` 命令

```
/memory list                              列出当前项目记忆
/memory add <title> <content>             添加
/memory get <title>                       读取
/memory search <keyword>                  全文搜索
/memory delete <title>                    删除
/memory stats                             统计
/memory cleanup <days>                    清理超过 N 天的记忆
```

底层映射到 `MemoryManager` 同名方法，由 `command.builtin.system.MemoryCommand` 实现。

## 6. 记忆与 LLM 协作

JHarness 默认**不会**自动注入记忆到系统提示词，因为：

- 记忆体量可能很大（数十 MB），全量注入会爆 token
- 注入策略与项目场景强相关，应由用户 / 插件按需决定

推荐玩法：

### 6.1 用 Hook 注入

写一个 `PRE_PROMPT` 类 hook（用 `USER_PROMPT_SUBMIT` 替代），从 `MemoryManager.searchMemories` 拿到相关条目，append 到 prompt：

```json
{
  "event": "user_prompt_submit",
  "type": "command",
  "command": "jharness-memory-injector.sh $ARGUMENTS"
}
```

### 6.2 用工具显式取

让 LLM 在需要时调用 `skill` 工具（或自定义的 `memory_search` 工具）按需取记忆，避免主动污染上下文。

## 7. 会话与记忆的区别再强调

| 维度 | 会话（SessionSnapshot） | 记忆（MemoryEntry） |
|------|------------------------|--------------------|
| **生命周期** | 一次连续对话 | 跨会话长期 |
| **粒度** | 整组消息 | 单条结构化条目 |
| **格式** | 完整 `ConversationMessage` 列表 | `title + content + category` |
| **存储位置** | `~/.jharness/data/sessions/<id>.json` | `~/.jharness/data/memories/<project>/<title>.json` |
| **自动注入对话上下文** | 否（用 `/resume` 主动恢复） | 否（除非用户 / 插件主动加载） |
| **管理命令** | `/session` `/resume` `/rewind` `/export` `/share` | `/memory` |
| **典型场景** | 接着上次的任务继续做 | 记住"项目编码规范"、"团队约定" |
