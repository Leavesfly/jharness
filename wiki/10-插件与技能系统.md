# 10 - 插件与技能系统

> 位于 `io.leavesfly.jharness.extension`。这是 JHarness 两大**非代码**扩展入口：Plugin（目录打包）与 Skill（Markdown）。

## 1. 技能系统（Skill）

### 1.1 概念

技能是一段带元数据的 **Markdown** 文本，用于"提示词复用 + 行为定义"。`SkillTool` 可在运行时调用。

### 1.2 数据结构

```java
public class SkillDefinition {
    final String name;         // 唯一名称
    final String description;  // 简介
    final String content;      // Markdown 全文（含 YAML front-matter）
    final String source;       // bundled / user / project / plugin
}
```

### 1.3 三层加载

`SkillLoader.loadSkillRegistry(Path cwd)` 按顺序合并三层来源：

| 层级 | 来源 | 源标签 |
|------|------|--------|
| 1. 内置 | classpath `resources/skills/` | `bundled` |
| 2. 用户 | `~/.jharness/skills/*.md` | `user` |
| 3. 项目 | `<cwd>/.jharness/skills/*.md` | `project` |

**同名覆盖策略**：后加载者覆盖前者（项目 > 用户 > 内置）。

### 1.4 内置技能目录结构

支持两种组织方式：

```
skills/
├── code-review/
│   └── SKILL.md                    # 子目录形式
└── quick-summary.md                # 扁平形式
```

从 JAR 加载时通过 `FileSystems.newFileSystem(jarUri, ...)` 创建 `jar:` FS，统一使用 `Files.list` API。

### 1.5 Skill Markdown 格式

```markdown
---
name: code-review
description: 代码审查助手
---

你是一个代码审查专家。请对提供的代码做以下检查：
1. 代码质量与可读性
2. 潜在 Bug 与安全问题
3. 性能优化建议
```

YAML front-matter 可选：若没有则 `name` 默认用文件名（去 `.md`），`description` 为空。

### 1.6 `SkillRegistry` API

```java
SkillRegistry reg = SkillRegistry.withBundled();          // 仅 bundled + user
SkillRegistry reg = SkillRegistry.withProject(cwd);       // 三层全开

reg.register(skill);
reg.get(name);
reg.getSkillNames();
reg.getAllSkills();
reg.size();
```

内部使用 `ConcurrentHashMap<String, SkillDefinition>`。

### 1.7 缓存优化

`SkillLoader` 使用 `volatile SkillRegistry cachedRegistry` + `volatile Path cachedProjectPath`：若再次以同一 `cwd` 调用，直接返回缓存，避免重复扫描文件系统。

### 1.8 在系统提示词中注入

`SystemPromptBuilder.withSkills(registry)` 会把所有技能的 `name: description` 追加到 `## 可用技能` 段：

```
## 可用技能
- code-review: 代码审查助手
- debug-strategy: 调试思路清单
```

### 1.9 路径工具类 `SkillPaths`

```java
SkillPaths.getUserSkillsDir();                // ~/.jharness/skills
SkillPaths.getBundledSkillsDir();             // 返回 null（classpath 无物理路径）
SkillPaths.getProjectSkillsDir(cwd);          // <cwd>/.jharness/skills
```

## 2. 插件系统（Plugin）

### 2.1 概念

Plugin 是一个**目录**，包含清单 + 可选的 Skills / Hooks / MCP 配置。是把一组扩展能力打包为单元的标准方式。

### 2.2 目录结构

```
my-plugin/                        # 插件根（也支持 .claude-plugin/ 子目录放 manifest）
├── plugin.json                   # 清单（必须）
├── skills/                       # 可选，默认目录名
│   └── my-skill.md
├── hooks.json                    # 可选
└── mcp.json                      # 可选
```

支持的清单路径（`PluginLoader.findManifest`）：
1. `<pluginDir>/plugin.json`
2. `<pluginDir>/.claude-plugin/plugin.json`

### 2.3 `PluginManifest` 字段

| 字段 | 默认 | 说明 |
|------|------|------|
| `name` | — | 插件名（目录名兜底） |
| `version` | — | 语义化版本 |
| `description` | — | 简介 |
| `enabledByDefault` | `true` | 未在 `enabledPlugins` 中显式配置时的行为 |
| `skillsDir` | `"skills"`（兜底扫描） | 技能目录（相对插件根） |
| `hooksFile` | — | Hook 配置文件 |
| `mcpFile` | — | MCP 配置文件 |

### 2.4 插件发现与加载

```
PluginLoader.discoverPluginPaths(cwd)
    ├─ ~/.jharness/plugins/*          （PluginPaths.getUserPluginsDir）
    └─ <cwd>/.jharness/plugins/*      （PluginPaths.getProjectPluginsDir）

PluginLoader.loadPlugins(settings, cwd)
    ├─ 对每个目录 loadPlugin(...)
    ├─ 基于 settings.enabledPlugins 判断启用
    └─ 返回 List<LoadedPlugin>
```

### 2.5 `LoadedPlugin` 运行时结构

```java
class LoadedPlugin {
    PluginManifest manifest;
    Path path;
    boolean enabled;
    List<SkillDefinition> skills;                      // 从 skills/ 或 manifest.skillsDir 加载
    Map<String,List<Object>> hooks;                    // 按事件分组的 Hook 原始对象
    Map<String,Object> mcpServers;                     // 整块 mcp.json 内容
    List<SkillDefinition> commands;                    // 预留（当前版本未使用）
}
```

### 2.6 启用 / 禁用

`Settings.enabledPlugins`（`Map<String, Boolean>`）：

```json
{
  "enabledPlugins": {
    "my-plugin":    true,
    "legacy-hooks": false
  }
}
```

查找逻辑：`enabledPlugins.getOrDefault(name, manifest.isEnabledByDefault())`。

### 2.7 `PluginInstaller`：安装 / 卸载

```java
PluginInstaller.installPlugin(Path source);     // 复制 → ~/.jharness/plugins/{name}
PluginInstaller.uninstallPlugin(String name);   // 删除
```

安全检查：
- 插件名不得含 `/`, `\`, `..`, 空名
- 目标路径必须位于 `userPluginsDir` 内（`startsWith` 检查），防穿越
- 已存在时先递归删除再复制

文件拷贝使用 `Files.walkFileTree` + `SimpleFileVisitor`，删除使用反向排序 `Stream<Path>`（先文件后目录）。

### 2.8 插件提供的三类扩展

#### 2.8.1 Skills
从 `<pluginDir>/skills/` 与 `<pluginDir>/<manifest.skillsDir>/` 两处加载（都扫描，结果合并），标记 `source="plugin"`。

#### 2.8.2 Hooks
读取 `<pluginDir>/<manifest.hooksFile>` 的 JSON，结构为 `Map<String, List<Object>>`（事件名 → Hook 定义列表）：

```json
{
  "pre_tool_use": [
    { "type": "command", "command": "echo about to run", "matcher": "bash" }
  ],
  "post_tool_use": [
    { "type": "http", "url": "https://audit.example.com/hook" }
  ]
}
```

JHarness 上层在 `createFullRegistry` 时把这些原始对象反序列化成 `HookDefinition` 子类注册到 `HookRegistry`。详见 [11](11-Hook系统.md)。

#### 2.8.3 MCP
读取 `<pluginDir>/<manifest.mcpFile>`：

```json
{
  "mcpServers": {
    "fs-server": { "type": "stdio", "command": "npx", "args": ["-y", "@modelcontextprotocol/server-filesystem"] }
  }
}
```

被 `McpClientManager.addServer` 批量加载，详见 [12](12-MCP协议客户端.md)。

### 2.9 `/plugin` 命令

由 `ConfigPlusCommandHandler.createPluginCommand` 注册：

- `/plugin list` — 列出已发现插件
- `/plugin enable <name>` / `/plugin disable <name>`
- `/reload-plugins` — 重新扫描 & 装载

## 3. 端到端示例：自制 `git-helper` 插件

```
~/.jharness/plugins/git-helper/
├── plugin.json
├── skills/
│   └── commit-msg.md
└── hooks.json
```

`plugin.json`：
```json
{
  "name": "git-helper",
  "version": "1.0.0",
  "description": "Git 协助插件",
  "enabledByDefault": true,
  "hooksFile": "hooks.json"
}
```

`skills/commit-msg.md`：
```markdown
---
name: commit-msg
description: 生成符合 Conventional Commits 的提交消息
---
你是提交消息生成助手。请按 `<type>(<scope>): <subject>` 格式生成 commit message……
```

`hooks.json`：
```json
{
  "post_tool_use": [
    {
      "type": "command",
      "matcher": "bash",
      "command": "grep -q 'git commit' <<< \"$OPENHARNESS_HOOK_PAYLOAD\" && echo '已执行提交'",
      "timeout_seconds": 5
    }
  ]
}
```

启动 JHarness 后：
- `SkillRegistry` 自动加载 `commit-msg`
- `SkillTool` 即可调用该技能
- 每次 `bash` 工具调用完成触发 post_tool_use Hook
