# 08 - 权限系统

> 位于 `io.leavesfly.jharness.session.permissions`。JHarness 的安全栅栏，所有工具调用 / 后台任务 / MCP 子进程 fork 前都会经过它。

## 1. 组件全景

| 类 | 职责 |
|----|------|
| `PermissionMode` | 三级模式枚举 |
| `PermissionChecker` | 核心评估器，线程安全 |
| `PermissionDecision` | `allow` / `deny` / `requiresConfirmation` |
| `PathRule` | 单条路径规则（pattern + allow） |

## 2. 三级权限模式

```java
public enum PermissionMode { DEFAULT, FULL_AUTO, PLAN }
```

| 模式 | 只读操作 | 写入 / 执行 | 典型场景 |
|------|---------|-------------|---------|
| **DEFAULT** | 直接放行 | 需要用户确认 | 日常交互 |
| **FULL_AUTO** | 放行 | 放行（但仍受黑名单/路径规则约束） | CI / 脚本、批处理 |
| **PLAN** | 放行 | **一律拒绝** | 计划 / 只读审查 |

⚠️ **关键安全语义**：`FULL_AUTO` 的含义是"不弹确认框"，**不是"放弃安全校验"**。工具/命令黑名单、路径 deny 规则在所有模式下都强制生效。

## 3. 评估流程（严格顺序）

```
evaluate(toolName, readOnly, filePath, command)
    │
    ▼
┌─────────────────────────────────────────┐
│ 1. deniedTools.contains(toolName)       │ → DENY
├─────────────────────────────────────────┤
│ 2. command 命中 deniedCommands 模式     │ → DENY
│    - 原字符串尝试匹配                    │
│    - 归一化后（tokenize + 去引号）再匹配 │
├─────────────────────────────────────────┤
│ 3. filePath 命中 pathRules              │
│    - realPath 解 symlink                │
│    - normalize 字符串兜底               │
│    - deny 规则 → DENY                   │
│    - allow 规则 → pathAllowHit=true     │
├─────────────────────────────────────────┤
│ 4. toolAllowListed = allowedTools.has(?) │
├─────────────────────────────────────────┤
│ 5. mode switch                          │
│    FULL_AUTO  → ALLOW                   │
│    PLAN       → readOnly ? ALLOW : DENY │
│    DEFAULT    → readOnly || pathAllowHit │
│                || toolAllowListed        │
│                   ? ALLOW                │
│                   : CONFIRM              │
└─────────────────────────────────────────┘
    │
    ▼
audit(toolName, readOnly, filePath, command, decision)
```

## 4. 命令黑名单：抗变形设计

简单的字符串匹配无法阻挡攻击者，`PermissionChecker` 做了两层归一化：

### 4.1 `normalizeCommand` 归一化规则

```
rm${IFS}-rf /       →  rm -rf /
'r'm -rf /          →  rm -rf /
rm\ -rf\ /          →  rm -rf /
```

做法：
- 剥离引号（`'` `"`）
- 折叠空白（`\s+` → 单空格）
- 展开 `${IFS}` 为普通空格
- 去除 `\` 转义

### 4.2 模式匹配

`deniedCommandPatterns` 使用 glob 风格（`globToRegex` 转换），例：

| pattern | 命中的命令 |
|---------|-----------|
| `rm -rf *` | `rm -rf /`、`rm -rf /tmp`、`rm -rf node_modules` |
| `sudo *` | `sudo apt update`、`sudo rm` |
| `curl * \| sh` | `curl https://x \| sh`（管道命令） |

## 5. 路径规则：抗穿越设计

### 5.1 多重检查

```java
for (PathRule rule : pathRules) {
    boolean hit = rule.matches(filePath)                    // 原始路径
               || rule.matches(normalizeForRuleMatch(...))  // 规范化路径
               || (realPath != null && rule.matches(realPath));  // realPath
    ...
}
```

### 5.2 `resolveRealPath` 细节

```java
Path p = Paths.get(filePath).toAbsolutePath().normalize();
// 已存在 → toRealPath()（跟随符号链接，解 ../）
// 不存在 → 父路径 realPath + filename
```

这样可防御：
- `../../../../etc/passwd` 式路径穿越
- 在项目内创建 symlink 指向 `/etc/shadow` 的 TOCTOU 攻击

### 5.3 规则优先级

```json
{
  "pathRules": [
    { "pattern": "src/**",      "allow": true },
    { "pattern": "**/*.env",    "allow": false },
    { "pattern": "/etc/**",     "allow": false }
  ]
}
```

**顺序扫描，首个命中生效**（deny 立即返回，allow 置位后继续走模式决策）。

## 6. 工具白名单与 PLAN 模式的正交性（FP-6）

白名单仅能跳过"用户二次确认"，**不能绕过 PLAN 模式的写阻断**：

```
mode == PLAN 且 非只读操作
    → 即便 toolAllowListed == true 也拒绝
```

否则"允许写某目录"会变成 PLAN 模式的逃逸通道。

## 7. 线程安全（FP-15）

```java
private volatile PermissionMode mode;                         // 可见性
private final List<PathRule>   pathRules      = new CopyOnWriteArrayList<>();
private final Set<String>      deniedCommands = new CopyOnWriteArraySet<>();
private final Set<String>      allowedTools   = new CopyOnWriteArraySet<>();
private final Set<String>      deniedTools    = new CopyOnWriteArraySet<>();
```

**读多写少** 场景下 CoW 结构完美适配：`evaluate` 高频读，`addXxx` / `setMode` 通常在启动时或 `/permissions` 命令时调用。

## 8. 审计日志（FP-8）

每次 `evaluate` 调用独立 logger：

```java
private static final Logger auditLogger = LoggerFactory.getLogger("jharness.permission.audit");
```

审计行字段：

```
[audit] tool=bash readOnly=false path=null cmd="rm -rf /tmp/build..." decision=DENY reason=命令被拒绝: rm -rf /tmp/build
```

在 `logback.xml` 中可为 `jharness.permission.audit` 配置独立 appender 落盘：

```xml
<logger name="jharness.permission.audit" level="INFO" additivity="false">
  <appender-ref ref="PERMISSION_AUDIT_FILE"/>
</logger>
```

长 path / cmd 由 `shorten(value, N)` 截断，避免日志爆炸。

## 9. 跨组件的 Checker 共享（FP-3）

在 `JHarnessApplication#buildQueryEngine`：

```java
PermissionChecker checker = new PermissionChecker(mode);
// 装配白/黑名单、路径规则、命令黑名单
...
// 关键：所有可 fork 子进程的入口都注入同一个 checker
taskManager.setPermissionChecker(checker);
mcpManager.setPermissionChecker(checker);
```

这样确保：
- `BashTool` 走 checker（前台）
- `BackgroundTaskManager.createShellTask` fork 前走 checker
- `McpClientManager.connectStdio` 启动子进程前走 checker
- `HookExecutor` Command Hook fork 前走 checker

**同一套规则全线守护**。

## 10. `PermissionDecision` 结构

```java
class PermissionDecision {
    enum Type { ALLOW, DENY, CONFIRM }
    Type type;
    String reason;

    static allow();
    static deny(String reason);
    static requiresConfirmation(String reason);
    boolean isAllow() / isDeny() / isConfirm();
}
```

DEFAULT 模式下 `CONFIRM` 的处理由 UI 层完成：
- `ConsoleInteractiveSession` 打印「是否执行 ... ? [y/N]」
- `TerminalUI` 可弹出模态确认框
- 脚本模式 `-p` 下 `CONFIRM` 视作拒绝（安全兜底）

## 11. 配置示例

`~/.jharness/settings.json`：

```json
{
  "permissionMode": "DEFAULT",
  "allowedTools": ["read_file", "grep", "glob"],
  "deniedTools":  ["bash"],
  "pathRules": [
    { "pattern": "src/**",      "allow": true  },
    { "pattern": "**/*.env*",   "allow": false },
    { "pattern": "**/secrets/**","allow": false },
    { "pattern": "/etc/**",     "allow": false },
    { "pattern": "~/.ssh/**",   "allow": false }
  ],
  "deniedCommandPatterns": [
    "rm -rf *",
    "sudo *",
    "chmod 777 *",
    "* | sh",
    "curl * | bash",
    "eval *"
  ]
}
```

## 12. 运行时切换

```
/permissions              # 查看当前模式与规则
/permissions plan         # 切到 PLAN
/permissions full_auto    # 切到 FULL_AUTO
```

底层调用 `permissionChecker.setMode(...)`，因 `volatile` 即刻对所有正在评估的线程可见。
