# 08 · 权限系统

> 包路径：`io.leavesfly.jharness.capability.permission`
>
> 核心类：`PermissionChecker`（编排者） + `PermissionRule` 接口 + 4 个责任链规则

---

## 1. 设计目标

JHarness 的工具能读写文件、执行 Shell、调用网络服务，因此需要一套**显式、可审计、不可绕过**的权限系统：

- **显式**：每个工具调用都要走一次 `PermissionChecker.evaluate()`
- **可审计**：所有决策记到 `jharness.permission.audit` 专属 logger
- **不可绕过**：即使 `FULL_AUTO` 模式也要过黑名单、路径规则
- **可扩展**：规则用责任链组装，业务侧可注入自定义 `PermissionRule`
- **SPI 隔离**：`kernel` 只看 `PermissionGate` 接口，不依赖具体实现

---

## 2. 三种权限模式

```java
public enum PermissionMode { DEFAULT, FULL_AUTO, PLAN }
```

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| **`DEFAULT`** | 只读自动放行；写入 / 执行前需要用户确认（除非路径白名单 / 工具白名单命中） | 默认日常使用 |
| **`FULL_AUTO`** | 所有操作放行（**黑名单 / 路径规则依然生效**） | CI / 后台跑批 |
| **`PLAN`** | 仅允许只读；所有写入操作一律拒绝（路径白名单也不能绕过） | "让 AI 先出方案" |

切换方式：

- CLI：`--permission-mode full_auto`
- 配置文件：`{"permissionMode": "FULL_AUTO"}`
- 运行时：`/permissions full_auto` 命令
- 编程：`permissionChecker.setMode(PermissionMode.FULL_AUTO)`

模式切换会写一行审计日志：`[PERM] mode_changed: DEFAULT -> FULL_AUTO`。

## 3. 责任链评估

`PermissionChecker.evaluate(toolName, isReadOnly, filePath, command)` 按顺序遍历 4 个规则：

```
DeniedToolsRule  →  DeniedCommandsRule  →  PathRulesRule  →  ModeDecisionRule
   (工具黑名单)        (命令黑名单)            (路径规则)         (模式决策)
```

**第一个产出非空 `PermissionDecision` 的规则即为终态**；若 4 个都返回 `Optional.empty()`，兜底放行（理论上不会发生，因为 `ModeDecisionRule` 永远会产出决策）。

### 3.1 规则一：`DeniedToolsRule`

如果 `ctx.deniedTools.contains(toolName)` → `deny("工具 xxx 在黑名单中")`，否则放行让下游评估。

### 3.2 规则二：`DeniedCommandsRule`

仅对 `command` 字段非空的调用生效（典型是 `bash` 工具）：

1. 用 `CommandPatternMatcher.normalize(command)` 规范化（去引号 / 把 `$IFS` 替换为空格 / 合并空白）
2. 用 `CommandPatternMatcher.matches(normalized, pattern)` 做 **glob 匹配**（仅支持 `*` 和 `?`，自动转义其他正则元字符）
3. 命中 → `deny("命令被黑名单拒绝: " + pattern)`

默认黑名单（`defaults/settings.json`）：

```json
"deniedCommandPatterns": [
  "rm -rf /*",  "rm -rf ~",     "rm -rf $HOME*",
  "sudo rm *",  "mkfs*",         "dd if=*",
  ":(){ :|:& };:",
  "curl * | sh",  "wget * | sh"
]
```

### 3.3 规则三：`PathRulesRule`

对 `filePath` 字段非空的调用生效：

1. `PathNormalizer.normalize(filePath)` → 普通路径归一
2. `PathNormalizer.resolveRealPath(filePath)` → 解符号链接（抗 TOCTOU / 穿越）
3. 用三种形式分别尝试匹配每条 `PathRule`：
   - 命中 `allow=false` → 立即 `deny`
   - 命中 `allow=true` → 标记 `ctx.markPathAllowHit()`，**不放行**，继续后续规则
4. 都不命中 → 放行让下游评估

**关键语义**：路径白名单命中只是"标记可跳过用户确认"，并不能让 `PLAN` 模式下的写操作通过（详见 § 3.4）。

默认路径黑名单（`defaults/settings.json`）：

```json
"pathRules": [
  { "pattern": "**/.git/**",         "allow": false },
  { "pattern": "**/.ssh/**",         "allow": false },
  { "pattern": "**/.aws/**",         "allow": false },
  { "pattern": "**/.kube/**",        "allow": false },
  { "pattern": "**/node_modules/**", "allow": false },
  { "pattern": "**/target/**",       "allow": false },
  { "pattern": "**/build/**",        "allow": false },
  { "pattern": "**/.env*",           "allow": false }
]
```

### 3.4 规则四：`ModeDecisionRule`（链路终点）

| `mode` | 行为 |
|--------|------|
| `FULL_AUTO` | `allow()` |
| `PLAN` + `readOnly=true` | `allow()` |
| `PLAN` + `readOnly=false` | `deny("计划模式阻止所有写入操作")` |
| `DEFAULT` + `readOnly=true` | `allow()` |
| `DEFAULT` + 路径白名单命中 | `allow()` |
| `DEFAULT` + 工具白名单命中 | `allow()` |
| `DEFAULT` + 写操作 | `requiresConfirmation("非只读操作需要确认")` |

## 4. `PermissionDecision`

```java
public class PermissionDecision {
    public static PermissionDecision allow();
    public static PermissionDecision deny(String reason);
    public static PermissionDecision requiresConfirmation(String reason);
}
```

| 决策 | UI 行为 | 工具结果 |
|------|---------|----------|
| `allow` | 直接执行 | 正常 `ToolResult` |
| `deny` | 不执行 | `ToolResult.error("权限被拒绝: " + reason)` |
| `requiresConfirmation` | 弹确认框（CLI：[Y/n]，TUI：模态） | 用户允许 → 执行；拒绝 → 同 `deny` |

## 5. 自定义规则

注入自定义规则链（适合二次开发场景）：

```java
List<PermissionRule> rules = List.of(
    new DeniedToolsRule(),
    new DeniedCommandsRule(),
    new MyCompanySecretRule(),   // 自定义：禁止访问 secrets.yaml
    new PathRulesRule(),
    new ModeDecisionRule()
);

PermissionChecker checker = new PermissionChecker(PermissionMode.DEFAULT, rules);
```

实现 `PermissionRule`：

```java
public class MyCompanySecretRule implements PermissionRule {
    @Override
    public Optional<PermissionDecision> evaluate(PermissionContext ctx) {
        if (ctx.getFilePath() != null && ctx.getFilePath().endsWith("secrets.yaml")) {
            return Optional.of(PermissionDecision.deny("禁止访问 secrets.yaml"));
        }
        return Optional.empty();  // 不决策，交下游
    }
}
```

## 6. 审计日志

所有决策通过 `PermissionAuditor.audit(...)` 写到独立 logger `jharness.permission.audit`，便于在 `logback.xml` 中分离到独立文件：

```
[PERM] tool=bash ro=false mode=DEFAULT decision=confirm reason="非只读操作需要确认" path="" cmd="git status"
[PERM] tool=write_file ro=false mode=DEFAULT decision=deny reason="路径 /etc/passwd 被规则拒绝: /etc/**" path="/etc/passwd" cmd=""
[PERM] tool=read_file ro=true mode=DEFAULT decision=allow reason="" path="src/Main.java" cmd=""
[PERM] mode_changed: DEFAULT -> FULL_AUTO
```

`shorten(str, 200)` 自动截断超长字段，防止日志爆炸。

## 7. 命令模式匹配细节

`CommandPatternMatcher.normalize`：

| 输入 | 输出 |
|------|------|
| `'rm' -rf "/tmp"` | `rm -rf /tmp` |
| `rm${IFS}-rf${IFS}/tmp` | `rm -rf /tmp` |
| `rm\trf\n/tmp` | `rm rf /tmp` |
| `rm   -rf  /tmp` | `rm -rf /tmp` |

`globToRegex`：

| Glob | Regex |
|------|-------|
| `rm -rf *` | `\Qrm -rf \E.*` |
| `sudo *` | `\Qsudo \E.*` |
| `curl * | sh` | `\Qcurl \E.*\Q | sh\E` |

字面量段用 `\Q...\E` 转义，避免用户在 pattern 中输入正则元字符引发注入。

---

## 8. 路径归一化细节

`PathNormalizer`：

- `normalize(p)` → `Paths.get(p).normalize().toString()`（去 `..` 和 `.`）
- `resolveRealPath(p)` → `Paths.get(p).toRealPath()`（解符号链接，文件不存在则返回 `null`）

**抗 TOCTOU**：检查时用 `realPath` 评估，而非用户传入的原始字符串，防止符号链接欺骗。

## 10. 与工具系统的协作

工具实现要正确传递 `filePath` / `command` 字段给权限系统：

```java
// FileEditTool 示例
@Override
public CompletableFuture<ToolResult> execute(FileEditToolInput input, ToolExecutionContext ctx) {
    PermissionDecision decision = ctx.getPermissionChecker()
        .evaluate(getName(), isReadOnly(input), input.file_path, null);
    if (!decision.isAllowed()) {
        return CompletableFuture.completedFuture(
            ToolResult.error("权限被拒绝: " + decision.getReason()));
    }
    // ... 实际编辑逻辑
}
```

实际上 `ToolCallDispatcher` 已经统一在调度层做了一次权限检查，工具内部无需重复（但工具应正确实现 `isReadOnly`）。

## 11. SPI 抽象 `PermissionGate`

```java
package io.leavesfly.jharness.kernel.spi;

public interface PermissionGate {
    PermissionDecision evaluate(String toolName, boolean readOnly, String filePath, String command);
    void setMode(PermissionMode mode);
    PermissionMode getMode();
    void addPathRule(String pattern, boolean allow);
    void addDeniedCommand(String pattern);
    void addAllowedTool(String toolName);
    void addDeniedTool(String toolName);
}
```

`PermissionChecker implements PermissionGate`，让 `kernel` / `integration.mcp` / `capability.task` / `capability.hook` 都只看接口、不绑死实现。
