# 17 · 扩展开发指南

本章汇总 JHarness 的 6 种主要扩展方式，每种给出端到端的代码示例与最佳实践。

---

## 1. 自定义工具

适用场景：让 LLM 能调用一个新的内部服务、查询某个数据库、做某个领域计算。

### 1.1 步骤

1. 在 `tools/input/<cat>/` 写一个输入 DTO（Jakarta Validation 注解）
2. 在 `tools/builtin/<cat>/` 继承 `BaseTool<T>`
3. 注册到 `ToolRegistry`

### 1.2 示例：`internal_user_query` 工具

```java
// io.leavesfly.jharness.tools.input.internal.InternalUserQueryInput
package io.leavesfly.jharness.tools.input.internal;

import jakarta.validation.constraints.NotBlank;

public class InternalUserQueryInput {
    @NotBlank
    public String userId;

    public boolean includeAvatar = false;
}
```

```java
// io.leavesfly.jharness.tools.builtin.internal.InternalUserQueryTool
package io.leavesfly.jharness.tools.builtin.internal;

import io.leavesfly.jharness.tools.*;
import io.leavesfly.jharness.tools.input.internal.InternalUserQueryInput;
import java.util.concurrent.CompletableFuture;

public class InternalUserQueryTool extends BaseTool<InternalUserQueryInput> {
    private final UserService userService;

    public InternalUserQueryTool(UserService userService) {
        this.userService = userService;
    }

    @Override public String getName() { return "internal_user_query"; }
    @Override public String getDescription() {
        return "查询内部员工信息。输入员工 ID，返回姓名、部门、邮箱。includeAvatar=true 时附带头像 URL。";
    }
    @Override public Class<InternalUserQueryInput> getInputClass() { return InternalUserQueryInput.class; }
    @Override public boolean isReadOnly(InternalUserQueryInput input) { return true; }

    @Override
    public CompletableFuture<ToolResult> execute(InternalUserQueryInput in, ToolExecutionContext ctx) {
        return CompletableFuture.supplyAsync(() -> {
            try {
                User u = userService.findById(in.userId);
                StringBuilder sb = new StringBuilder();
                sb.append("姓名: ").append(u.getName()).append("\n");
                sb.append("部门: ").append(u.getDepartment()).append("\n");
                sb.append("邮箱: ").append(u.getEmail());
                if (in.includeAvatar) {
                    sb.append("\n头像: ").append(u.getAvatarUrl());
                }
                return ToolResult.success(sb.toString());
            } catch (Exception e) {
                return ToolResult.error("查询失败: " + e.getMessage());
            }
        });
    }
}
```

### 1.3 注册

修改 `app.bootstrap.QueryEngineBuilder.build()`，在 `ToolRegistry.withDefaults(...)` 之后：

```java
registry.register(new InternalUserQueryTool(userService));
```

### 1.4 最佳实践

| 项 | 建议 |
|----|------|
| **`getName`** | 全小写 + 下划线，全局唯一；建议加业务前缀避免与内置/MCP 冲突 |
| **`getDescription`** | 写"何时用 + 何时不用 + 输入要求"；LLM 完全依赖描述决策 |
| **`isReadOnly`** | 实事求是。错误标记只读会导致 PLAN 模式下执行写操作 |
| **DTO 字段命名** | 用 snake_case，与 JSON Schema 风格一致 |
| **`execute` 返回** | 强制异步：CPU 密集 → `supplyAsync`；IO → 用自己的 ExecutorService |
| **错误处理** | 用 `ToolResult.error("xxx")`，绝不抛出未捕获异常 |
| **超时控制** | 内部用 `Future.get(timeout)` 或 `CompletableFuture.orTimeout` |

详细参考 → [06-工具系统](06-工具系统.md)

## 2. 自定义斜杠命令

适用场景：用户在 REPL 里需要一个本地动作（不经过 LLM）。

### 2.1 简单实现

```java
SlashCommand cmd = new SimpleSlashCommand(
    "wc",
    "统计当前消息历史的字数",
    (args, ctx, ec) -> {
        int total = ctx.getEngine().getMessages().stream()
            .flatMap(m -> m.getContent().stream())
            .filter(b -> b instanceof TextBlock)
            .mapToInt(b -> ((TextBlock) b).getText().length())
            .sum();
        return CompletableFuture.completedFuture(
            CommandResult.success("消息历史共 " + total + " 字"));
    }
);

commandRegistry.register(cmd);
```

### 2.2 复杂实现

参考 `command/builtin/system/MemoryCommand.java`、`command/builtin/config/ConfigCommandHandler.java`：

- 多子命令分发（`add` / `list` / `delete` / ...）
- 调用领域 Manager 完成业务
- 返回结构化 `CommandResult.message`（可含 Markdown）

详细参考 → [07-斜杠命令系统](07-斜杠命令系统.md)

## 3. 自定义 LLM Provider

适用场景：接入非 OpenAI 协议的 LLM（如 Anthropic 原生、Gemini、自家私有协议）。

### 3.1 步骤

1. 实现 `kernel.spi.LlmGateway` 接口
2. 在 `QueryEngineBuilder.build()` 中替换 `apiClient`

### 3.2 示例骨架

```java
package com.mycompany.jharness.api;

import io.leavesfly.jharness.kernel.engine.model.ConversationMessage;
import io.leavesfly.jharness.kernel.engine.stream.StreamEvent;
import io.leavesfly.jharness.kernel.spi.LlmGateway;
import io.leavesfly.jharness.integration.api.ApiMessageCompleteEvent;

import java.util.List;
import java.util.Map;
import java.util.concurrent.CompletableFuture;
import java.util.function.Consumer;

public class MyAnthropicClient implements LlmGateway {

    @Override
    public CompletableFuture<ApiMessageCompleteEvent> streamMessage(
            List<ConversationMessage> messages, String systemPrompt,
            List<Map<String, Object>> tools, Consumer<StreamEvent> eventConsumer) {
        // 1. 把 messages 转换为 Anthropic 原生格式
        // 2. POST 到 https://api.anthropic.com/v1/messages
        // 3. 解析 SSE 流，逐块推 AssistantTextDelta
        // 4. 累计文本 + tool_uses → 包装为 ApiMessageCompleteEvent
        // 5. 返回 CompletableFuture
        return ...;
    }

    @Override public void cancelAllActiveRequests() { /* OkHttp dispatcher.cancelAll() */ }
    @Override public String getProviderName() { return "anthropic"; }
    @Override public String getModelName() { return "claude-3-5-sonnet"; }
}
```

### 3.3 替换装配

修改 `QueryEngineBuilder.build()` 中创建 apiClient 的那段：

```java
LlmGateway apiClient = "anthropic".equals(settings.getProvider())
    ? new MyAnthropicClient(...)
    : new OpenAiApiClient(...);

QueryEngine engine = new QueryEngine(apiClient, toolRegistry, permissionChecker,
                                      cwd, systemPrompt, maxTurns);
```

**QueryEngine 完全不依赖具体实现**，无需改任何内核代码。

详细参考 → [05-API客户端 § 13](05-API客户端.md#13-自定义-provider)

## 4. 集成自定义 MCP 客户端

适用场景：自己实现一个 MCP 服务器并希望被 JHarness 当工具调用。

### 4.1 stdio 方式

1. 实现一个能讲 JSON-RPC 2.0 的命令行程序（任意语言）
2. 在 `~/.jharness/mcp-servers.json` 加配置：

```json
{
  "mcpServers": {
    "my-tools": {
      "type": "stdio",
      "command": "/usr/local/bin/my-mcp-server",
      "args": ["--config", "/etc/my-mcp.yaml"]
    }
  }
}
```

3. 启动 JHarness，工具会自动出现在 `/mcp tools` 列表中

### 4.2 HTTP 方式

```json
{
  "mcpServers": {
    "remote-tools": {
      "type": "http",
      "url": "https://my-mcp.example.com/mcp",
      "headers": { "Authorization": "Bearer ${MY_TOKEN}" }
    }
  }
}
```

环境变量 `${MY_TOKEN}` 由 JHarness 启动时自动展开（仅在配置加载阶段）。

### 4.3 协议约束

服务器必须实现以下 JSON-RPC 方法：

- `initialize` — 握手 + 能力协商
- `tools/list` — 返回工具清单
- `tools/call` — 调用工具
- `resources/list` — （可选）返回资源清单
- `resources/read` — （可选）读资源

`tools/list` 返回的每个工具必须含 `name` / `description` / `inputSchema`（标准 JSON Schema）。

详细参考 → [12-MCP协议客户端](12-MCP协议客户端.md)

## 5. 自定义 Hook

适用场景：在生命周期节点注入审计 / 通知 / 自动化逻辑。

### 5.1 Command Hook

`~/.jharness/hooks.json`：

```json
{
  "session_start": [
    {
      "type": "command",
      "command": "/usr/local/bin/notify",
      "args": ["JHarness 启动: cwd=$CWD model=$MODEL"],
      "timeoutMs": 3000
    }
  ],
  "stop": [
    {
      "type": "command",
      "command": "/usr/local/bin/log-cost.sh",
      "args": ["$SESSION_ID", "$TURNS"]
    }
  ]
}
```

### 5.2 HTTP Hook

```json
{
  "user_prompt_submit": [
    {
      "type": "http",
      "url": "https://my-logger.example.com/log",
      "timeoutMs": 2000,
      "blocking": false
    }
  ]
}
```

### 5.3 Prompt Hook（注入系统提示词）

```json
{
  "user_prompt_submit": [
    {
      "type": "prompt",
      "prompt": "请用中文回答，并在末尾附上参考资料。"
    }
  ]
}
```

### 5.4 Java Subprocess Hook

```json
{
  "session_end": [
    {
      "type": "java",
      "mainClass": "com.example.MyHook",
      "classpath": "/opt/hooks/my-hook.jar",
      "args": ["--mode", "cleanup"]
    }
  ]
}
```

### 5.5 注意

- `JHARNESS_HOOK_DEPTH` 防递归：最大 10 层
- Command Hook 与权限闸门集成：PLAN 模式下不执行写命令
- HTTP Hook 走 SSRF 防护

详细参考 → [11-Hook系统](11-Hook系统.md)

## 6. 自定义插件

适用场景：把"技能 + 命令 + Hook + MCP 服务器 + 子代理"打包分发。

### 6.1 目录结构

```
my-plugin/
├── plugin.json
├── skills/
│   └── code-review.md
├── commands/
│   └── review.md
├── agents/
│   └── senior-reviewer.md
├── hooks.json
└── mcp-servers.json
```

### 6.2 `plugin.json`

```json
{
  "name": "my-plugin",
  "version": "1.0.0",
  "description": "示例插件",
  "enabledByDefault": true,
  "skillsDir": "skills",
  "commandsDir": "commands",
  "agentsDir": "agents",
  "hooksFile": "hooks.json",
  "mcpFile": "mcp-servers.json"
}
```

### 6.3 安装

把整个 `my-plugin/` 目录放到 `~/.jharness/plugins/` 或 `<cwd>/.jharness/plugins/`，重启 / `/reload-plugins` 即生效。

也可放到 `~/.claude/plugins/` 自动被回退识别（Claude Code 兼容）。

### 6.4 兼容 Claude Code 插件

JHarness 的 `PluginManifest` 用 `@JsonIgnoreProperties(ignoreUnknown = true)`，能直接读 Claude Code 生态的插件清单。多数情况下把 Claude Code 插件目录复制过来即可工作；不识别的字段（如 Claude 特有的某些配置）会被静默忽略。

详细参考 → [10-插件与技能系统](10-插件与技能系统.md)

## 7. 自定义子代理

适用场景：让主 Agent 在合适时机派发子任务给"专家角色"。

### 7.1 定义角色

`~/.jharness/agents/code-reviewer.md`：

```markdown
---
name: code-reviewer
description: 严格的代码审查员
---

你是一位严格的资深 Java 工程师，专注于：
- 代码可读性
- 并发安全
- 性能瓶颈
- 单元测试覆盖

回答时逐条列出问题，给出具体修改建议，并标注严重程度（High/Medium/Low）。
```

### 7.2 LLM 调用

主 Agent 看到上下文需要审查时，自动通过 `agent_spawn`：

```json
{
  "tool_calls": [{
    "function": {
      "name": "agent_spawn",
      "arguments": "{\"name\":\"code-reviewer\",\"task\":\"审查 src/Main.java\"}"
    }
  }]
}
```

### 7.3 主动并行调用

也可由用户在命令中显式触发：

```
/agents create review-team
/agents add code-reviewer review-team
/agents add senior-reviewer review-team
（主 Agent 自动派发任务给团队）
```

详细参考 → [13-多智能体协调](13-多智能体协调.md)

## 8. 自定义权限规则

适用场景：公司内部有"机密文件 / 高危命令"特殊规则。

### 8.1 实现 `PermissionRule`

```java
package com.mycompany.jharness.permission;

import io.leavesfly.jharness.capability.permission.*;
import java.util.Optional;

public class CompanySecretRule implements PermissionRule {
    @Override
    public Optional<PermissionDecision> evaluate(PermissionContext ctx) {
        String path = ctx.getFilePath();
        if (path == null) return Optional.empty();

        if (path.endsWith("secrets.yaml") || path.contains("/credentials/")) {
            return Optional.of(PermissionDecision.deny("禁止访问凭据文件: " + path));
        }
        if (path.matches(".*/customer-data/.*\\.csv")) {
            return Optional.of(PermissionDecision.requiresConfirmation(
                "访问客户数据需要人工确认: " + path));
        }
        return Optional.empty();   // 不决策，交下游
    }
}
```

### 8.2 装配

修改 `QueryEngineBuilder.build()` 中创建 `PermissionChecker` 的部分：

```java
List<PermissionRule> rules = List.of(
    new DeniedToolsRule(),
    new DeniedCommandsRule(),
    new CompanySecretRule(),      // ← 自定义规则
    new PathRulesRule(),
    new ModeDecisionRule()
);

PermissionChecker checker = new PermissionChecker(mode, rules);
```

详细参考 → [08-权限系统 § 5](08-权限系统.md#5-自定义规则)

## 10. 代码风格 / 测试 / CI

### 10.1 风格

- Java 17 语法 + 模式匹配 / Records / Switch Expression
- 命名：类名 PascalCase，方法 / 字段 camelCase，常量 UPPER_SNAKE，包路径全小写
- 包路径分层（详见 [03-整体架构](03-整体架构.md)），禁止反向依赖
- 注释：复杂逻辑必须有 Javadoc；TODO 必须带 owner 和 ticket 号
- Jackson：复用 `JacksonUtils.MAPPER` / `PRETTY_MAPPER` 单例

### 10.2 测试

- 单元测试：JUnit 5 + AssertJ + Mockito
- 集成测试：覆盖关键链路（API 客户端 / 权限 / 引擎 / 工具）
- 推荐用真实小模型（Ollama 本地 qwen3.5:4b）做端到端 smoke 测试

### 10.3 CI

`mvn -DskipTests=false clean test`。建议在 PR 模板中要求：

- 新工具必须有对应 JUnit 测试
- 新规则必须有单元测试（包括边界 / 注入 / 路径穿越）
- 新 Hook Runner 必须有超时 / 错误处理测试

## 11. SPI 接口速查

| SPI | 包 | 用途 |
|-----|-----|------|
| `LlmGateway` | `kernel.spi` | LLM 客户端抽象 |
| `ToolCatalog` | `kernel.spi` | 工具目录抽象 |
| `PermissionGate` | `kernel.spi` | 权限闸门抽象 |
| `BaseTool<T>` | `tools` | 工具基类 |
| `SlashCommand` | `command` | 斜杠命令接口 |
| `PermissionRule` | `capability.permission` | 权限规则 |
| `HookRunner` | `capability.hook.runtime` | Hook 执行器 |
| `McpSession` | `integration.mcp.session` | MCP 会话 |
| `QueryEngineFactory` | `capability.coordination.AgentOrchestrator` | 子代理引擎工厂 |
| `CronJobExecutor` | `integration.cron.CronRegistry` | 定时作业执行器 |

## 12. 关键文件位置

| 扩展类型 | 装配点（推荐修改位置） |
|---------|----------------------|
| 新工具 | `tools/ToolRegistry#withDefaults` 或 `QueryEngineBuilder` |
| 新命令 | `command/CommandRegistry#registerAllCommands` |
| 新 LLM Provider | `app.bootstrap.QueryEngineBuilder#build` |
| 新权限规则 | `app.bootstrap.QueryEngineBuilder#build` |
| 新 MCP 服务器 | 配置文件 `mcp-servers.json`（不改代码） |
| 新 Hook | 配置文件 `hooks.json`（不改代码） |
| 新插件 | 直接放目录（不改代码） |
| 新子代理 | `~/.jharness/agents/<name>.md`（不改代码） |
