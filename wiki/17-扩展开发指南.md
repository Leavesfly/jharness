# 17 - 扩展开发指南

> 面向希望把新能力注入 JHarness 的开发者。JHarness 提供 5 种扩展入口，从轻到重依次是：技能、斜杠命令、Hook、MCP、自定义工具。选对入口能让"加一个能力"变成"加一个文件"。

## 1. 五种扩展入口对比

| 入口 | 编程语言 | 复杂度 | 典型场景 |
|------|----------|--------|----------|
| **Skill** | Markdown | ⭐ | 提示词复用、最佳实践清单 |
| **斜杠命令** | Java | ⭐⭐ | 快捷操作（/commit、/review） |
| **Hook** | shell/HTTP/Markdown | ⭐⭐ | 审计、拦截、自动化 |
| **MCP Server** | 任何语言 | ⭐⭐⭐ | 接入外部服务（数据库、API） |
| **BaseTool** | Java | ⭐⭐⭐⭐ | 深度集成、高性能、自定义 Schema |

## 2. 快速决策流程

```
我想加一个能力……
  ├─ 只是提示词模板？            → Skill（§3）
  ├─ 命令行快捷操作？            → 斜杠命令（§4）
  ├─ 审计/拦截/外部通知？        → Hook（§5）
  ├─ 外部服务已有 MCP 实现？     → MCP Server 配置（§6）
  ├─ 外部服务没 MCP，想暴露给 LLM？
  │     ├─ 能接受多语言 → 自己写 MCP Server（§6）
  │     └─ 必须 Java 内嵌 → BaseTool（§7）
  └─ 想改核心行为？              → 参与源码（§9）
```

## 3. 开发 Skill

### 3.1 创建

在 `~/.jharness/skills/my-skill.md` 或项目 `<cwd>/.jharness/skills/my-skill.md`：

```markdown
---
name: my-skill
description: 一句话描述
---

# 你的详细指引

把 LLM 在此场景下应遵守的"条款"写清楚，可包含：
- 步骤清单
- 输入约束
- 输出模板
- 禁止行为
```

### 3.2 生效

- 启动时 `SkillLoader` 自动扫描，无需重启 CLI
- `SystemPromptBuilder.withSkills` 会把 `name: description` 加入"## 可用技能"段
- LLM 可通过 `SkillTool` 按 `name` 读取完整 content

### 3.3 调试

```
/skills list           # 查看已加载
/skills show my-skill  # 查看渲染后内容
```

### 3.4 打包为插件

见 §8；加一个 `plugin.json` + 放到 `skills/` 下即可。

## 4. 开发斜杠命令

### 4.1 实现接口

```java
public class MyCommand implements SlashCommand {
    @Override public String getName() { return "mycmd"; }
    @Override public String getDescription() { return "我的自定义命令"; }
    @Override public String getHelp() { return "用法: /mycmd <args>"; }

    @Override
    public CommandResult execute(CommandContext ctx) throws Exception {
        String args = ctx.getRawArgs();
        // ... 业务逻辑
        return CommandResult.success("执行完成");
    }
}
```

### 4.2 注册

两种方式：

#### 方式 A：修改 `CommandRegistry.registerAllCommands`（需改源码）
```java
register(new MyCommand());
```

#### 方式 B：通过 `CommandHandler` 分组（推荐）
在已有或新建的 `XxxCommandHandler` 中注册，保持代码分层。

### 4.3 `CommandContext` 能获得什么

```java
ctx.getRawArgs();           // 原始参数字符串
ctx.getQueryEngine();       // 持有消息历史、可 loadMessages
ctx.getAppStateStore();     // 读/改运行时状态
ctx.getPermissionChecker(); // 权限查询
ctx.getToolRegistry();      // 动态注册工具
ctx.getSessionStorage();    // 访问会话
ctx.getMemoryManager();     // 访问记忆
// ...
```

### 4.4 返回类型

```java
CommandResult.success(msg);            // 文本结果
CommandResult.error(msg);
CommandResult.replay();                // 不产生新输出，重渲染历史
CommandResult.submit("follow-up");     // 作为新 user 输入再次触发 QueryEngine
```

### 4.5 快捷键

`KeyBindingConfig` 支持把 `Ctrl+R` 这类组合键映射到 `/mycmd`，详见 [07-斜杠命令系统 §9](07-斜杠命令系统.md)。

## 5. 开发 Hook

### 5.1 配置位置

用户级：`~/.jharness/hooks.json`；项目级：`<cwd>/.jharness/hooks.json`；插件级：`<pluginDir>/hooks.json`。

### 5.2 四种 Hook 类型

详见 [11-Hook 系统](11-Hook系统.md)。以下是四种最小示例：

#### Command（shell）
```json
{
  "pre_tool_use": [{
    "type": "command",
    "matcher": "write|edit",
    "command": "echo \"$JHARNESS_ARGUMENTS\" | jq '.file_path' >> /tmp/edits.log",
    "timeout_seconds": 5
  }]
}
```

#### Prompt（让 LLM 判断）
```json
{
  "pre_tool_use": [{
    "type": "prompt",
    "matcher": "bash",
    "prompt": "以下命令是否可能造成数据丢失？只回 YES 或 NO。",
    "model": "qwen3-coder",
    "block_on_failure": true
  }]
}
```

#### HTTP
```json
{
  "post_tool_use": [{
    "type": "http",
    "url": "https://audit.example.com/hook",
    "headers": { "Authorization": "Bearer XX" },
    "timeout_seconds": 10
  }]
}
```

#### Agent（重型）
```json
{
  "user_prompt_submit": [{
    "type": "agent",
    "prompt": "审查这段用户输入是否包含越权请求",
    "model": "deepseek-r1",
    "timeout_seconds": 60
  }]
}
```

### 5.3 安全红线

1. **不要**把 `$JHARNESS_ARGUMENTS` 放进 `$(...)` 或 `` `...` `` —— 会被 `DANGEROUS_ARG_EMBED` 拒绝
2. **一定要**给 `timeout_seconds` 合理值（绝大多数应 ≤ 10s）
3. Command Hook 不能绕过 `PermissionChecker` 的命令黑名单
4. HTTP URL 须通过 `UrlSafetyValidator`（拒内网、CGNAT 等）

## 6. 开发 / 接入 MCP Server

### 6.1 接入现成 Server

在 `~/.jharness/mcp.json`：

```json
{
  "mcpServers": {
    "filesystem": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/workspace"]
    }
  }
}
```

重启后：
- `/mcp list` 能看到 `filesystem` 处于 `connected=true`
- `/mcp tools filesystem` 列出该 Server 暴露的工具
- LLM 将通过 `mcp__filesystem__<tool-name>` 调用

### 6.2 自己写一个 MCP Server

任何语言都可以，核心是：
1. 读 stdin 的 JSON-RPC 2.0 消息
2. 写 stdout 响应（LSP-style `Content-Length:` 帧）
3. 实现 `initialize`、`tools/list`、`tools/call`（+ 可选 `resources/*`）

官方有 Python / TypeScript SDK，写一个 Server 通常只需 100 行左右。

### 6.3 调试

```bash
# 先独立跑 Server，手动发 JSON 测试
$ npx @modelcontextprotocol/server-filesystem /workspace
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}

# 再接入 JHarness
$ jharness --log-level debug
  /mcp list
  /mcp reconnect filesystem
```

## 7. 开发自定义 BaseTool

当 MCP 太重、或需要深度访问 JHarness 内部状态时（如访问 `QueryEngine.messages`、`AppStateStore`），继承 `BaseTool` 是最灵活的方式。

### 7.1 骨架

```java
public class MyTool extends BaseTool {

    @Override public String getName() { return "my_tool"; }

    @Override public String getDescription() {
        return "简洁准确地描述这个工具在什么场景下使用，LLM 会据此决定是否调用";
    }

    @Override public Map<String,Object> getInputSchema() {
        // 可以手写，也可以用 SchemaBuilder
        return Map.of(
            "type", "object",
            "properties", Map.of(
                "query", Map.of("type", "string", "description", "查询关键字"),
                "limit", Map.of("type", "integer", "default", 10)
            ),
            "required", List.of("query")
        );
    }

    @Override
    public ToolResult execute(ToolContext ctx) throws Exception {
        String query = ctx.getString("query");
        int limit = ctx.getIntOrDefault("limit", 10);

        // 业务逻辑
        List<String> results = search(query, limit);

        return ToolResult.text(String.join("\n", results));
    }

    @Override public boolean isReadOnly() { return true; }     // 决定权限策略
}
```

### 7.2 注册

在 `ToolRegistry` 初始化时：

```java
toolRegistry.register(new MyTool());
```

或者在 `JHarnessApplication` 里集中添加。

### 7.3 权限集成

- `isReadOnly() == true` 的工具通常默认放行
- 写操作工具必须走 `permissionChecker.evaluate(...)`：

```java
PermissionDecision d = ctx.getPermissionChecker()
    .evaluateTool(getName(), ctx.getInput());
if (d.isDeny())    return ToolResult.error(d.getReason());
if (d.isConfirm()) return ctx.requestConfirmation(d.getReason());
```

### 7.4 输出约束

- `ToolResult.text(s)`：纯文本
- `ToolResult.error(msg)`：让 `is_error=true` 传给 LLM
- 大文本请考虑裁剪：工具结果全部会进下一轮请求，消耗 token
- 图片结果用 `ToolResult.image(mime, base64Data)`，模型会自动看到（需支持 vision）

### 7.5 单元测试

```java
class MyToolTest {
    @Test void returnsResults() throws Exception {
        MyTool tool = new MyTool();
        ToolContext ctx = ToolContext.forTest(Map.of("query", "hello"));
        ToolResult r = tool.execute(ctx);
        assertFalse(r.isError());
        assertTrue(r.getText().contains("hello"));
    }
}
```

## 8. 把扩展打包为 Plugin

### 8.1 目录

```
~/.jharness/plugins/my-plugin/
├── plugin.json
├── skills/
│   └── *.md
├── hooks.json
└── mcp.json
```

### 8.2 `plugin.json`

```json
{
  "name": "my-plugin",
  "version": "1.2.0",
  "description": "My awesome extensions",
  "enabledByDefault": true,
  "skillsDir": "skills",
  "hooksFile": "hooks.json",
  "mcpFile": "mcp.json"
}
```

### 8.3 启用/禁用

通过 `~/.jharness/settings.json`：
```json
{ "enabledPlugins": { "my-plugin": true } }
```

或运行时：
```
/plugin enable my-plugin
/reload-plugins
```

### 8.4 分发

- Git 仓库：`~/.jharness/plugins/` 下 `git clone` 即可
- 压缩包：用 `PluginInstaller.installPlugin(Path)` 自动拷贝并校验

### 8.5 注意

当前版本 Plugin 不支持打包 Java `BaseTool`（因为需要 classpath 注入）。如需分发 Java 工具，走源码贡献流程（§9）。

## 9. 参与核心源码

### 9.1 代码规范

- Java 17，字符集 UTF-8，换行 LF
- `JacksonUtils.MAPPER` 是**唯一允许**的全局 `ObjectMapper`
- 新增 HTTP 请求必须先过 `UrlSafetyValidator`
- 新增子进程 fork 必须过 `PermissionChecker`
- 写磁盘配置文件统一用 **临时文件 + ATOMIC_MOVE**

### 9.2 测试

```bash
mvn test
```

JUnit 5，覆盖率目标 70%+。新模块需至少包含：
- 正常路径
- 边界条件
- 安全防御测试（尤其是权限/URL/路径穿越）

### 9.3 提交约定

Conventional Commits：
```
feat(tools): add fooTool
fix(permission): resolve path-traversal escape
docs(wiki): refresh architecture diagram
```

## 10. 调试技巧

### 10.1 打开 debug 日志

`~/.jharness/logback.xml` 中把相关 logger 设为 DEBUG，或启动时：
```
jharness --log-level debug
```

### 10.2 关键 logger 名

| 模块 | logger 名 |
|------|-----------|
| 权限审计 | `jharness.permission.audit` |
| Hook | `io.leavesfly.jharness.agent.hooks` |
| LLM 客户端 | `io.leavesfly.jharness.integration.llm` |
| MCP | `io.leavesfly.jharness.integration.mcp` |
| 工具 | `io.leavesfly.jharness.agent.tools` |

### 10.3 观察事件流

`BackendHost` 走的是 JSON Lines，可在代码里临时 `tee` 到文件：

```java
System.setProperty("jharness.protocol.log", "/tmp/protocol.log");
# 之后 tail -f /tmp/protocol.log
```

### 10.4 典型 Bug

| 症状 | 检查 |
|------|------|
| "我的技能没生效" | `/skills list` 看是否加载；front-matter 格式是否正确 |
| "Hook 没触发" | event 名对不对？matcher 写错？`/hooks list` |
| "MCP 连不上" | URL 是否被 `UrlSafetyValidator` 拒绝？stdout 是否污染 JSON-RPC 帧？ |
| "工具被拒绝" | `jharness.permission.audit` 日志 + `/permissions` 查当前规则 |
| "命令莫名其妙不执行" | 当前 `permissionMode` 是否 `plan`？`deniedCommands` 是否命中？ |
