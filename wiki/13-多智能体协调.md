# 13 - 多智能体协调

> 位于 `io.leavesfly.jharness.agent.coordinator`（该包同时包含 `TeamRegistry`、`TeamRecord`、`AgentOrchestrator`）。JHarness 支持把复杂任务拆分给多个 Sub-Agent，在一个"团队"里协作。

## 1. 两种组织方式

| 模式 | 适用 | 实现 |
|------|------|------|
| **Team** | 长期存在、可互发消息的 Agent 组 | `TeamRegistry` + `TeamRecord` |
| **Orchestrator** | 单次任务拆分，并行/顺序调度 | `AgentOrchestrator` |

两者独立存在、可以组合。Orchestrator 更偏"编程调度"；Team 更偏"会话型协作"。

## 2. Team 子系统

### 2.1 数据模型

#### `AgentInfo`
```java
class AgentInfo {
    String agentId;
    String name;
    String role;               // 可选角色说明
    String model;              // 该 Agent 使用的模型
    Map<String,Object> config; // 扩展配置
}
```

#### `TeamMessage`
```java
class TeamMessage {
    String messageId;
    String fromAgent;          // 发送方 agentId
    String toAgent;            // 接收方 agentId（可为 "*" 广播）
    String content;
    Instant timestamp;
    Map<String,Object> metadata;
}
```

#### `TeamRecord`
```java
class TeamRecord {
    String teamId;
    String name;
    String description;
    List<AgentInfo> members;            // 成员列表
    List<TeamMessage> messageHistory;   // 消息历史
    Instant createdAt;
    Instant updatedAt;
    Map<String,Object> metadata;
}
```

### 2.2 `TeamRegistry` API

```java
TeamRegistry reg = new TeamRegistry(teamsDir);   // 默认 ~/.jharness/teams

String teamId = reg.createTeam(name, description);
reg.deleteTeam(teamId);
reg.addAgent(teamId, AgentInfo agent);
reg.removeAgent(teamId, agentId);

reg.sendMessage(teamId, TeamMessage msg);        // 追加到历史
List<TeamMessage> history = reg.getMessageHistory(teamId, limit);

TeamRecord team = reg.getTeam(teamId);
List<TeamRecord> all = reg.listTeams();
```

### 2.3 持久化

`TeamRegistry` 对每个 team 存一个文件：

```
~/.jharness/teams/team-{teamId}.json
```

写入策略与 `SessionStorage` 一致：**临时文件 + ATOMIC_MOVE**。

### 2.4 `/team` 命令

由 `TeamCommandHandler.createTeamCommand` 注册，典型子命令：

- `/team create <name>`
- `/team list`
- `/team add <teamId> <agent-name>`
- `/team send <teamId> <to> <content>`
- `/team history <teamId>`

### 2.5 典型使用场景

- **专家型团队**：`coder` + `reviewer` + `tester` 分工
- **多模型对比**：同一问题让不同 model 的 Agent 回答并聚合
- **领域协作**：`frontend-agent` 与 `backend-agent` 分别处理各自领域的任务

## 3. Orchestrator 子系统

### 3.1 设计目标

把一个大任务拆分成若干 **SubTask**，分发给 **SubAgent**，按策略（并行 / 顺序 / 有向图）执行并聚合结果。

### 3.2 核心类

| 类 | 作用 |
|---|------|
| `AgentOrchestrator` | 调度入口 |
| `SubAgent` | 子 Agent 抽象（持有独立 `QueryEngine`） |
| `SubTask` | 任务描述（prompt、依赖、超时） |
| `OrchestrationResult` | 结果聚合：`Map<taskId, SubAgentResult>` |
| `ExecutionStrategy` | 策略枚举：`PARALLEL` / `SEQUENTIAL` / `DAG` |

### 3.3 调度流程

```
AgentOrchestrator.execute(List<SubTask> tasks, strategy)
  ├─ PARALLEL:
  │    ExecutorService.invokeAll(tasks as Callable)
  │    → 并行触发每个 SubAgent 的 QueryEngine
  │
  ├─ SEQUENTIAL:
  │    for task in tasks: runOne(task); 结果注入下一任务 context
  │
  └─ DAG:
       拓扑排序后分批并行
       出度归零的节点触发执行
       所有节点完成或任何节点失败时结束
```

### 3.4 `SubAgent`

每个 SubAgent 都有独立的：
- `QueryEngine` 实例（独立的 message history）
- `systemPrompt`（可注入 role 描述）
- `toolRegistry`（可裁剪）
- `cwd`（通常与父一致）

但共享：
- `LlmProvider`（复用 OkHttp 连接池）
- `PermissionChecker`（统一权限）
- `CostTracker`（聚合 token 成本）

### 3.5 结果聚合

```java
OrchestrationResult {
    Map<String, SubAgentResult> results;     // taskId → result
    UsageSnapshot totalUsage;                // token 汇总
    Duration totalDuration;
    List<String> failedTaskIds;
}
```

失败策略：
- `PARALLEL`：单任务失败不影响其他，记录到 `failedTaskIds`
- `SEQUENTIAL`：前置失败终止后续（可配置为 continueOnFailure）
- `DAG`：失败节点的下游自动跳过

## 4. Orchestrator 与工具联动

Orchestrator 可以被注册为一个 **工具**（`OrchestratorTool`），让父 Agent 通过正常的 `tool_use` 调用：

```
父 Agent → tool_use: orchestrator
          args: { tasks: [...], strategy: "parallel" }
          → SubAgent × N 并发
          → 结果聚合 → 返回父 Agent
```

这让多 Agent 协作对 LLM 来说**完全透明**，与调用任何其他工具一致。

## 5. 后台任务挂钩（`TaskRecord`）

多 Agent 执行常常是长耗时任务，JHarness 用 `TaskRecord` 做后台任务表示：

```java
class TaskRecord {
    String taskId;
    TaskType type;        // LOCAL_BASH / LOCAL_AGENT / REMOTE_AGENT / IN_PROCESS
    String description;
    TaskStatus status;    // PENDING / RUNNING / COMPLETED / FAILED / STOPPED / KILLED
    Instant createdAt;
    Instant startedAt;
    Instant endedAt;
    String output;
    Map<String,Object> metadata;
}
```

- `LOCAL_BASH`：本地 shell 作业（见 [14](14-后台任务与Cron.md)）
- `LOCAL_AGENT`：本地 fork 的 JHarness 子会话
- `REMOTE_AGENT`：通过 `BridgeSessionManager` 远端执行
- `IN_PROCESS`：进程内线程（Orchestrator 的 SubAgent 多用此类型）

Orchestrator 为每个 SubTask 创建一条 `TaskRecord`，通过 `/tasks` 命令可查询实时状态。

## 6. `/agent`、`/team`、`/delegate` 命令

典型命令链（来自 `CommandRegistry`）：

- `/agent list` — 本地预置的 Agent 模板
- `/agent spawn <template>` — 临时起一个 SubAgent
- `/delegate <agent> <prompt>` — 把一段 prompt 委派出去执行
- `/team ...` — 见 §2.4

## 7. 取消与隔离

- **取消传播**：父 `QueryEngine` 取消时，会触发 `Orchestrator.cancelAll`，逐个 `SubAgent.cancel`
- **Token 限流**：每个 SubAgent 可以单独设 `maxOutputTokens`，防止单个子任务吞掉整个预算
- **异常隔离**：SubAgent 内部未捕获异常不会导致父进程崩溃，而是标记该 Task 为 `FAILED` 并继续
- **日志隔离**：每个 SubAgent 的日志 MDC 注入 `subAgentId`，方便检索

## 8. 实战示例

### 8.1 并行代码评审

```java
List<SubTask> tasks = List.of(
    new SubTask("reviewer-security",   "审查安全隐患"),
    new SubTask("reviewer-perf",       "审查性能问题"),
    new SubTask("reviewer-style",      "审查代码风格")
);
OrchestrationResult r = orchestrator.execute(tasks, ExecutionStrategy.PARALLEL);
```

### 8.2 串行文档生成

```
SEQUENTIAL:
  Step 1 (outliner)  → 生成大纲
  Step 2 (writer)    → 填充章节
  Step 3 (reviewer)  → 校对润色
```

### 8.3 DAG 特性开发

```
DAG:
  design  ──►  impl-backend  ──►  test
           └─► impl-frontend ──►  test
```
